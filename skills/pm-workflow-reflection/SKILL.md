# pm-workflow-reflection

PM 专用 skill——反思。入口：直接执行本文步骤；产出报告写入 `.pm/reflections/`。

## 触发规则

| 触发器        | 行为                             |
|---------------|----------------------------------|
| 用户说 `反思` | 启动本工作流（按当前模式选路径） |

## 语言要求

| 输出类型            | 语言         |
|---------------------|--------------|
| 反思报告 / 记忆文件 | **简体中文** |
| SKILL 注释          | **简体中文** |

## 路径裁决

| 当前模式 | 路径                                  |
|----------|---------------------------------------|
| 管理模式 | [管理模式反思](#管理模式下的反思)     |
| 闲聊模式 | [闲聊模式下的反思](#闲聊模式下的反思) |

## 闲聊模式下的反思

1. 读取 `.pm/user_profile.md`，逐维度检查：
   - 是否有与实际不符的条目（偏差）？
   - 是否有明显过时的信息（漂移）？
   - 表达是否可以更流畅、更自然、更贴合用户的喜好（优化）？
2. 读取 `.pm/chats/INDEX.md` + 最近 1-2 次聊天记录：
   - 对话风格是否与用户的偏好一致？
   - 是否有可以改进的互动模式？
3. 输出反思报告：
   - 发现的问题
   - 建议修正
   - 具体行动项
4. **落盘反思报告**：写入 `.pm/reflections/YYYY-MM-DD_HH-MM_session.md`，格式见下方模板

## 管理模式下的反思

1. 读取 `.pm/project_memory.md`，逐区检查：
   - Operational Conventions：近期操作是否全部遵循约定？
   - Recurring Project Context：Phase 状态、测试数、lint 状态等声称是否与代码实际一致？
   - Tech Decisions：是否有已过时或需补充的决策？
2. 读取 `docs/development_log.md` 索引 + 最近 3 条 devlog 条目：
   - 近期动作是否有执行失败未跟进的？
   - 是否有决策未记录？
3. 审计**本次会话触发了哪些工作流**：
   - 严格按触发词命中：Status / Next / Finish / Reflection / Leisure / Check / Memo / Deploy / Bug，各几次
   - OC5 管道执行了多少次完整闭环（即使不是命名工作流）
   - 哪些工作流本应触发但未触发（如长时间管道后应主动 Check）
   - 会话中实际支配力最大的是命名工作流还是 OC5 管道
4. 回顾**本 session 对话轮次**：
   - 统计总轮次数
   - 按表格逐轮列出：轮次编号 / 用户发言摘要 / PM 动作摘要 / 类型（status / dispatch / review / fix / plan / debug）
   - 分析模式：是否连续同类型操作、是否有反复修改同一文件的回合、是否有长时间等待 agent 回到的空转轮次
5. 输出反思报告：
   - 偏差 / 漂移发现
   - Prompt 遵循情况
   - 动作执行成功 / 失败复盘
   - 具体行动项
   - 文件读写是否可以优化 - 减少读写次数，或者改为更适合的时机（例如切换模式时）
   - 任务编排是否合理，是否过细或过粗，是否可以并行化 或 必须串行化但并行化了
   - agent 之间的 context 传递 是否可以优化，是 agent 通过 prompt 传递，还是通过记忆文件传递，或者两者结合？是否存在 context 漂移或过时未更新的情况？
   - agent 的行为是否符合预期，是否存在不符合预期的行为模式，是否需要调整 prompt 来修正行为，或者需要调整 OC 约定来引导行为
   - OC 约定是否需要调整，是否存在不合理或难以遵循的约定，是否需要新增约定来覆盖目前没有明确指导的情况
   - OC5 管道的执行是否符合预期，是否存在绕过管道的情况，是否需要调整管道配置来覆盖目前没有明确指导的情况
6. 输出**会话总结**（本次会话完成了哪些事情和动作）：
   - 按模块/主题分节
   - 每节列出关键动作 + 产出（PR #、文件变更、决策）
   - 末尾附关键数字（合入 PR 数、test 增量、新增约定数等）
7. 自问：**本次会话中有哪些可以固化为工作流？**
   - 是否出现了可复用的跨平台验证模式？
   - 是否有小修小补（注释/补测试/补签名）值得纳入 Finish 流程？
   - 是否有 agent 工具失败的排查链条值得写进反思？
   - 输出候选工作流列表（含一句话描述），由开发者裁决是否落笔
8. **落盘反思报告**：将完整的反思报告（含步骤 1-7 全部内容）写入 `.pm/reflections/YYYY-MM-DD_HH-MM_session.md`

## 报告格式

```markdown
# 反思报告 — YYYY-MM-DD

## 会话总结
| 模块 | 关键动作 | 产出 |
|------|----------|------|

## 偏差 / 漂移发现
| 问题 | 严重度 | 已修复 |
|------|--------|--------|

## 工作流触发统计
| 工作流 | 次数 | 备注 |
|--------|------|------|

## 对话轮次统计
> 总轮次：N（含本轮反思）

| # | 用户 | PM | 类型 |
|---|------|-----|------|
| 1 | `<摘要>` | `<动作>` | `<status/dispatch/review/fix/plan/debug>` |

**模式分析**：<连续同类型？反复改同一文件？长时间等 agent？>

## Agent 行为审计
| Agent | 派发次数 | 问题 | 处理 |
|-------|----------|------|------|

## 文件读写优化
- 高频文件及修改次数
- 改进空间

## 可固化的工作流
| 候选 | 描述 |
|------|------|

## OC 约定审查
| OC | 状态 | 本会话变动 |
|----|------|------------|
```

## 关键边界

- 落盘路径必须用 `.pm/reflections/YYYY-MM-DD_HH-MM_session.md`（分钟级时间戳 OC3.5）
- 步骤 6 候选工作流只是"输出候选"，**不**自动写入 OC 或 agent 文档——需开发者裁决
