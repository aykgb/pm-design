# ADR-002: Batch 规模阈值 — 2,000 行上限的经验依据

> 状态：Accepted  
> 日期：2026-06-13  
> 决策者：PM  
> 关联：`specs/spec-breakdown.md` §Batch 设计标准

## 背景

Spec 拆解 & Batch 设计标准定义了单 Batch 代码行数上限为 ~2,000 行。该阈值需要经验依据支撑，而非随意设定。

## 项目历史数据

以下为本项目（xidi-minimal）已完成的 Batch 与 Themis 审查结果：

| Batch | 行数（估） | Tests | Themis 首轮 | Fix 循环 |
|-------|-----------|-------|-------------|----------|
| P2 Batch A (T2→T1→T3→T8) | ~1,600 | +75 | 2P1 + 10P2 | R1→R2 |
| P2 Batch B (T4→T5→T6→T7) | ~1,100 | +53 | 0P0 / 0P1 / 3P2 | 无 |
| P2 e2e | ~200 | +5 | 0P0 | 无 |
| BL-PHASE2-SWEEP | ~300 | 0 | 0P0 | 无 |
| P3 Batch A (T1→T7) | ~1,600 | +60 | 少量 P0/P1 | R1→R2 |
| P3 Batch B (T8→T12) | ~1,400 | +58 | 待审查 | — |

## 观察

1. **~1,100 行是最优区间**：P2 Batch B 以 ~1,100 行实现 Themis 首轮 0P0，无 fix 循环。
2. **~1,600 行需 fix 循环**：P2 Batch A 和 P3 Batch A 均需 R1→R2，但 fix 规模小（通常 <200 行），仍可在一轮内收敛。
3. **<500 行几乎总是干净的**：P2 e2e 和 BL-PHASE2-SWEEP 均 0P0。
4. **非线性退化**：未测试 >2,000 行的 Batch，但从 1,100→1,600 已观察到审查质量下降（首轮 P0/P1 出现率从 0% 升至 >0%），推测 2,000+ 行 P0/P1 漏检率将显著上升。

## 决策

单 Batch 代码行数上限设为 **2,000 行**（净增+删），基于以下推理：

| 风险 | 说明 |
|------|------|
| **审查疲劳** | 2,000 行 ≈ 25-30 页 diff。Themis 单次审查注意力窗口有限，超出后 P0/P1 检出率下降 |
| **Fix loop 爆炸半径** | 审查 P0/P1 → 开发 fix → 重审（R2）。2,000 行 Batch 的 fix loop 投入是 1,000 行 Batch 的 ~2 倍 |
| **单文件膨胀** | `qmt_trade_worker.py` 已达 3,134 行。一个 2,000 行 Batch 若集中在此文件，单文件将超 5,000 行——接近维护性临界点 |
| **测试套件比例** | 2,000 行源码通常对应 50-75 个测试，是 QA 单次验证的合理上限 |

### 补充规则

- **下限不设硬约束**：<500 行的 Batch 允许但需避免过度拆分（见 `specs/spec-breakdown.md` 反模式：过度拆分）
- **上限非绝对**：若 Batch 内 task 共享同一 causal chain 且无法拆分（如全部修改 `qmt_trade_worker.py` 的同一条 submit→callback 链路），可放宽至 2,500 行，但需 PM 显式裁决
- **跨文件假内聚禁止**：不同 causal chain 的文件不能为"凑 Batch"而合并——这不会降低行数，只会增加 Themis 跳转审查的遗漏概率

## 后果

### 正面

- 审查质量可预测：<2,000 行时 Themis 首轮 P0/P1 检出率可接受
- Fix loop 成本可控：单轮 fix 通常 <300 行
- 单文件增长受控：不会出现一个 PR 把核心文件从 3,000 行炸到 6,000 行

### 负面

- 需要更多 PM 判断来拆分 vs 合并（如两个 1,200 行 Batch vs 一个 2,400 行 Batch）
- 对于高度内聚的模块（如 `qmt_trade_worker.py`），拆分可能引入"部分实现"的中间状态

### 监控指标

- 跟踪每个 Batch 的 Themis 首轮 P0/P1 数量与行数的关系
- 当 1,500-2,000 行区间出现 >2 P0 时，考虑收紧上限
